**Overview**
mzdata is a Rust crate that centralizes mass spectrometry format handling, raw spectrum modeling, and signal pipelines for both CLI tools and libraries. The root `lib.rs` wires together the entry points that a caller relies on: `io` for format readers/writers, `spectrum` for data models, `params` for metadata handling, and the optional `mzsignal` integration that brings signal-processing helpers.

**IO inference & dispatch**
The `src/io` module is the gateway for every file that enters or leaves mzdata. `io::infer_format` roots the format detection and dispatch stack: `inference.rs` figures out `MassSpectrometryFormat`, `dispatch.rs` exposes `MZReader`, `MZReaderType`, and `MZReaderBuilder`, and `pipeline.rs` defines the `MassSpectrometryReadWriteProcess`/`Source`/`Sink` helpers used for chaining readers. `MZReader::open_path` uses these pieces to map a path or stream through gzip/mag readers into the concrete reader type that implements the shared IO traits.

**Spectrum modeling**
`src/spectrum` defines the canonical data contracts. `bindata` and `frame` translate raw binary arrays into `MultiLayerSpectrum` objects, `peaks.rs` and `chromatogram.rs` expose higher-level peak/continuum accessors, and `spectrum_types.rs` plus `scan_properties.rs` hold trait implementations for RT, MS level, and mobility access. These types are shared by IO readers to present a uniform `SpectrumLike` interface regardless of format.

**EIC data path**
`src/io/eic.rs` defines `EICQuery`, `ExtractedIonChromatogram`, and the `ExtractedIonChromatogramSource` trait that every reader implements. Queries are normalized via `PreparedEICQuery`, and `extract_eics_from_spectra` drives the default work loop: it temporarily switches the reader detail level to `DetailLevel::Lazy`, iterates spectra (`SpectrumSource::len`/`get_spectrum_by_index`), filters via mz/RT/mobility bounds, reads raw arrays or peak lists, and accumulates intensity sums with `sum_array_range`. `MZReader` simply forwards `extract_eic(s)` to the concrete reader sitting behind it, so format-agnostic callers can take a pathB and still rely on the shared EIC pipeline.

**TDF fast path**
The Bruker TDF reader under `src/io/tdf` layers metadata (`metadata.rs`, `constants.rs`), array mapping helpers (`arrays.rs`), and SQLite-backed chromatogram tables (`sql.rs`). `reader.rs` exposes `TDFSpectrumReader` which implements `ExtractedIonChromatogramSource` using two routes: the standard spectrum loop and a specialized `extract_eics_fast`. The fast path precomputes `PreparedTDFEICQuery` entries (bounding TOF and mobility slices via the `mz_converter`/`im_converter` metadata), caches the current tims frame, and walks the entry index so each query only scans the necessary portion of a frame. `sum_tdf_query` then converts TOF indices back to m/z, applies intensity thresholds, and feeds times/intensities into the `ExtractedIonChromatogram` results. This dedicated code is why TDF files can deliver accelerated EIC responses compared to iterating every stored spectrum.

**Docs & navigation**
The documentation is anchored by `README.md` and `CHANGELOG.md`, but `docs/reader_tutorial.md`, `docs/writer_tutorial.md`, and the massive `docs/spectrum_tutorial.md` walk through opening readers, writing files, and interacting with the `MultiLayerSpectrum` types. New contributors can also explore `examples/` for runnable code snippets and `.planning/codebase` (this folder) for future maps. Refer to `src/io` comments to see how features like `mgf`, `mzmlb`, `thermo`, `proxi`, `imzml`, and `tdf` are gated by cargo features before diving in.
