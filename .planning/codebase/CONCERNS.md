## EIC extraction surface
- `src/io/eic.rs` defines `EICQuery`, `ExtractedIonChromatogram`, and an `ExtractedIonChromatogramSource` trait that sums spectra via `extract_eics_from_spectra`. The implementation is eager and reads every spectrum in a reader while assuming `raw_arrays` are filled for the target format; none of the `#[cfg(feature = ...)]` paths for MGF, mzML, mzMLb, or Thermo are exercised by unit tests, so regressions in any of those fast-path array-accessors will silently slip by until something tries them with real data. ([D:/Projects/mzdata/src/io/eic.rs](D:/Projects/mzdata/src/io/eic.rs#L1-L410))
- The Bruker-specific `TDFFrameReaderType` exposes only the fast-path `extract_eics_fast` helper (`src/io/tdf/reader.rs#L672-L707`) and `TDFSpectrumReaderType::extract_eics` simply proxies to `self.frame_reader.extract_eics_fast` without ever falling back to a manual per-scan summation or providing a finer-grained MS2/DIA treatment. That means any new EIC usage that hits PASEF/DIAPASEF frames relies on that fast path to correctly honor isolation-window boundaries, mobility ranges, and MS level filtering, yet we do not have an independent manual reference for those cases inside the same code path. This mismatch is still present and is a risk when adding the new EIC features. ([D:/Projects/mzdata/src/io/tdf/reader.rs#L672-L709](D:/Projects/mzdata/src/io/tdf/reader.rs#L672-L709), [D:/Projects/mzdata/src/io/tdf/reader.rs#L1322-L1324](D:/Projects/mzdata/src/io/tdf/reader.rs#L1322-L1324))

## Format dependencies & unknowns

([D:/Projects/mzdata/src/io/tdf/reader.rs#L1739-L1749](D:/Projects/mzdata/src/io/tdf/reader.rs#L1739-L1749))
- Several Bruker-related EIC/format tests are `#[ignore]` because they point at hard-coded desktop data under `C:\Users\ray\Desktop\MS_Data_20260318`. That means CI runners (and anyone without that directory) never execute the manual benchmark/smoke tests that are supposed to validate the fast-path extraction for real `.d` files, so regressions in the Bruker fast-sum code may be invisible until a developer has the exact dataset and un-ignores the tests. ([D:/Projects/mzdata/src/io/infer_format/mod.rs#L138-L506](D:/Projects/mzdata/src/io/infer_format/mod.rs#L138-L506))
- The new EIC trait is wired up to `mzMLb` and Thermo readers only under feature flags (`src/io/eic.rs#L381-L399`). Currently, no automated regression exercises these code paths, so fetching EICs from mzMLb/Thermo files is technically untested debt as long as those feature combinations remain off the default CI matrix. ([D:/Projects/mzdata/src/io/eic.rs#L381-L399](D:/Projects/mzdata/src/io/eic.rs#L381-L399))
- The manual extraction helpers in `src/io/infer_format/mod.rs` run a lazy loop over each spectrum and re-derive the same sums as the production code. They are useful for spotting divergence, but the helper itself is only used inside ignored, path-specific benchmarks, so the library lacks a portable ground truth for MS2/DIA EICs. ([D:/Projects/mzdata/src/io/infer_format/mod.rs#L57-L128](D:/Projects/mzdata/src/io/infer_format/mod.rs#L57-L128))

## CI/test limitations
- The `just test-units` shortcut runs `cargo nextest` with heaps of optional features (`nalgebra,parallelism,mzsignal,zlib-ng-compat,thermo,async,numpress,bruker_tdf`), which makes the suite heavy and long-running. Adding new EIC behavior increases pressure on that command, and until the workspace settles, rerunning it frequently may be frustrating for contributors without the same feature set. ([D:/Projects/mzdata/Justfile#L1-L17](D:/Projects/mzdata/Justfile#L1-L17))
- Coverage runs (`cargo llvm-cov --lib --tests nextest --features ... mzmlb,async,numpress`) do not enable the `bruker_tdf` feature, so the Bruker fast-path and any Bruker-specific EIC dispatch remain unmeasured in the coverage reports. That can hide regressions introduced by the new EIC module that only hit Bruker datasets. ([D:/Projects/mzdata/Justfile#L18-L28](D:/Projects/mzdata/Justfile#L18-L28))
- The `infer_format` regression harness also suffers from its ignored Bruker benchmarks, and the hard-coded dataset means you can't reproduce those benchmarks on CI, which leaves the EIC fast-path for `.d` files without automated verification after the recent changes. ([D:/Projects/mzdata/src/io/infer_format/mod.rs#L369-L506](D:/Projects/mzdata/src/io/infer_format/mod.rs#L369-L506))

## Unknowns / Technical debt
- It is still unclear how the EIC fast path handles MS2/DIA window boundaries when queries include both RT and mobility filters: the current fast path scans each frame, sums a block of scans and commits that block to every matching query, but there is no per-scan conversion layer or manual path to compare against, so subtle truncation bugs could exist until we add more precise sampling or regression data.
- The new `extract_eics_from_spectra` function assumes `spectrum.raw_arrays()` returns valid `m/z` and intensity arrays, yet some readers may only expose peak lists. In those cases we fall back to iterating peaks, but we still rely on the `SpectrumLike` implementation to provide sorted m/z arrays. If any of the optional readers are inconsistent with that expectation, the EIC sums will be wrong; without regression tests for `mzMLb`/Thermo, that remains a live unknown.
